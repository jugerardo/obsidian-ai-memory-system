# Import skill

When the user says **"import from inbox"**, **"process inbox"**, or **"import notes"**, run this procedure to triage files in `00_Inbox/`, generate or validate their frontmatter, and move them to the right folder.

This skill handles two modes:

- **Mode 1 — pre-formatted**: the file already has frontmatter (e.g. generated by the external AI wrap-up prompts). Validate, confirm, and move.
- **Mode 2 — raw**: the file has no frontmatter or is a raw export paste. Infer type, generate frontmatter, reformat body to match the right template, and present as a draft for review.

## Vault access

All reads, writes, and moves must go through the **obsidian-mcp**. Do not use file system tools (Read, Write, Edit, Bash) to touch vault files.

Key operations and their obsidian-mcp equivalents:

| What you need to do | obsidian-mcp tool to use |
|---|---|
| Read an existing note | `read_note` (or equivalent get/fetch tool) |
| Create a new note | `create_note` (or equivalent write tool) |
| Delete a note | `delete_note` (or equivalent) |
| List notes in a folder | `list_notes` / `list_files` |

If the obsidian-mcp is not connected or returns an error, **stop and tell the user** — do not fall back to file system tools silently.

## Inputs

- Files in `00_Inbox/` with a `.md` extension.
- The vault root.
- Today's date in `YYYY-MM-DD` (use the system date — do not guess).

## Step 1: Find inbox files

Use obsidian-mcp `list_notes` on `00_Inbox/`. Filter to `.md` files only.

- If the inbox is empty, tell the user "Nothing in the inbox to import" and stop.
- If there is one file, proceed directly to step 2.
- If there are multiple files, list them and ask: "Process all, or pick specific ones?" Proceed with the user's selection.

## Step 2: For each file — detect mode

Use obsidian-mcp `read_note` to read the file.

- **Mode 1**: file has a YAML frontmatter block with a `type:` field present. Proceed to step 3.
- **Mode 2**: file has no frontmatter, or frontmatter is missing `type:`. Proceed to step 4.

## Step 3 (Mode 1): Validate and prepare

Check that all required frontmatter fields are present and valid:

| Field | Required value |
|---|---|
| `type` | one of: `conversation`, `code-session`, `project-session` |
| `date` | valid `YYYY-MM-DD` date |
| `status` | `raw` (reset to `raw` if anything else — inbox files are unprocessed by definition) |
| `summary` | non-empty string |

Fill in any missing fields:

- `date`: use today's date if absent.
- `status`: set to `raw`.
- `source`: leave as-is if present; set to `other` if absent.
- `summary`: if absent, generate a one-sentence summary from the note title and body.

Then proceed to step 5 (determine destination).

## Step 4 (Mode 2): Infer type and reformat

### 4a. Infer note type

Read the full content and decide which type best fits:

- **`code-session`**: the content is primarily about writing, debugging, or exploring code. Strong signals: code blocks, error messages, library names, commands.
- **`project-session`**: the content covers substantive work on a named project — decisions, progress, outcomes. Strong signals: project name references, goal/outcome framing, multiple work areas.
- **`conversation`**: everything else — discussion, brainstorming, research, Q&A without code.

If the content is ambiguous between `project-session` and `conversation`, default to `conversation` and flag the uncertainty for the user.

### 4b. Generate frontmatter

Build the frontmatter block:

- `type`: from step 4a.
- `date`: extract from the content if a date is mentioned; otherwise use today.
- `source`: infer from content clues (e.g. "ChatGPT", "Gemini") if present; otherwise `other`.
- `status`: `raw`.
- `tags`: leave empty — the user or extraction skill can fill these.
- `summary`: generate a one-sentence description from the content.
- `project`: leave empty for now — step 5 handles project assignment.

### 4c. Reformat body

Restructure the content to match the appropriate template (`conversation.md`, `code-session.md`, or `project-session.md`). Map the raw content to the template's sections as best you can. Mark any sections you couldn't confidently fill with `<!-- could not infer — please fill in -->`.

Do not invent content. If a section has no corresponding material in the raw file, leave it blank rather than guessing.

## Step 5: Determine destination

### Routing rules

| Type | Default destination |
|---|---|
| `conversation` | `10_Conversations/<source>/YYYY-MM-DD_<slug>.md` |
| `code-session` (no project) | `30_Code_Sessions/YYYY-MM-DD_<slug>.md` |
| `code-session` (with project) | `20_Projects/<ProjectName>/sessions/YYYY-MM-DD_<slug>.md` |
| `project-session` | `20_Projects/<ProjectName>/sessions/YYYY-MM-DD_<slug>.md` |

### Project assignment

For `project-session` notes, and for `code-session` notes where the `project:` field is non-empty or the content references a project name:

1. Use obsidian-mcp `list_notes` on `20_Projects/` to get the list of active project folders.
2. For each project folder, use obsidian-mcp `read_note` to read its `CLAUDE.md` and extract the **Project name** from the "Project identity" section.
3. If the note's `project:` field is already set, suggest it as the default. Otherwise suggest the closest match based on content.
4. Present the list and ask the user to confirm: "This looks like it belongs to **[suggested project]**. Confirm, or pick a different one?" Also offer "None — save as standalone" to route a code session to `30_Code_Sessions/` instead.

For `code-session` notes with an empty `project:` field and no project references in the content, ask once: "Does this code session belong to a project?" — Yes triggers the project selection above; No routes to `30_Code_Sessions/`.

### Slug generation

Generate a kebab-case slug of ≤ 6 words from the note title or summary. Prefix with the date: `YYYY-MM-DD_<slug>.md`.

## Step 6: Present draft for review

Do **not** move or save anything yet. Present a summary for the user to confirm:

```
File: <inbox filename>
Mode: pre-formatted | inferred
Type: <type>
Destination: <full destination path>

Frontmatter:
<yaml block>

[Inferences made — Mode 2 only]:
- Type inferred as <type> because: <reason>
- Date taken from: <source>
- Project matched to: <project name> because: <reason>

[Action: Confirm / Edit / Skip]
```

For Mode 2, highlight every inference so the user can catch mistakes before the file is moved.

Wait for the user's decision per file. Apply any edits before proceeding.

## Step 7: Move file and update indexes

For each confirmed file:

1. Use obsidian-mcp `create_note` to write the final note (with validated/generated frontmatter and body) to the destination path.
2. Use obsidian-mcp `delete_note` to remove the original from `00_Inbox/`.
3. If routed to a project folder:
   - Use obsidian-mcp `read_note` to open the project's `_index.md`.
   - Use obsidian-mcp `update_note` to append a link under `## Recent sessions`: `- [[YYYY-MM-DD_slug|YYYY-MM-DD: short title]]`.

## Step 8: Reply

Reply with:

1. Number of files processed.
2. Number skipped (and why — user choice or error).
3. A list of destination paths for moved files.
4. A reminder: "Run the insight-extraction skill when ready to harvest insights from these notes."

## What this skill does NOT do

- It does **not** create insight notes — that's the insight-extraction skill's job.
- It does **not** modify files outside `00_Inbox/` and the destination folder + its `_index.md`.
- It does **not** delete inbox files until the user has confirmed the draft.
- It does **not** invent content for sections it can't fill — it leaves them blank and flags them.

## Edge cases

- **File has frontmatter but `type` is unrecognised**: treat as Mode 2 — ask the user what type it is.
- **Multiple files in inbox, user picks "process all"**: present all drafts before moving any of them, so the user can review everything in one pass.
- **Destination file already exists** (same date + slug): append `-2` to the slug rather than overwriting.
- **Project folder exists but has no `CLAUDE.md`**: include it in the project list anyway, noting that it has no identity file.
